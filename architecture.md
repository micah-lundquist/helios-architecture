# Helios Architecture

## §0. Why the harness exists

The context window is rebuilt fresh each turn from durable storage — messages, helios facts, summaries, spans — rather than carried forward as session state. This inversion is why the harness exists. Agents that hand themselves forward as session state behave like they've woken from a dream: each turn starts over, retrieves facts by similarity, and hopes they cohere with what the user just said. The harness treats the context window as a living document — what survives is written down, what was inferred is reconstructed. A corollary: the agent is a companion who also has hands. Subagent dispatch (§3 #6) lets the primary conversation continue uninterrupted while specialized work happens in isolated child contexts, with results flowing back as tool responses. The rest of this document describes the mechanics that realize this thesis.

**The key mechanical insight:** the backing model has no memory between turns. Every API call is independent — the model receives a freshly assembled prompt and responds as if the entire conversation happened in one sitting. Conversational awareness is an illusion maintained by the harness: all messages are permanently stored in SQLite, and the ContextAssembler rebuilds the prompt each turn from that storage, packing identity, memory, and as many recent messages as the per-agent context budget allows (configurable per agent). When messages exceed the budget, the classification-aware pruning ladder (§4 Trace D) selects which to keep: relational turns survive longest, noise drops first. Summaries capture the substance of pruned turns so older context degrades gracefully rather than disappearing.

## §1. What Helios is

Helios is a self-improving memory library for agents. It stores structured facts in SQLite, measures whether each fact actually influenced the model's response, classifies what kind of influence it had, and accumulates both signals into per-fact behavioral fingerprints that inform future injection decisions. Consumers wire Helios in as either an in-process Python library or a standalone HTTP server (see §2). The memory substrate is identical in both shapes.

The system's core problem is that storing facts is trivial but knowing which facts matter — and matter *how* — is not. A naive memory system retrieves by similarity and hopes for the best. Helios retrieves by similarity, injects the top candidates into the model's context, then measures what happened. Over time, each fact accumulates a behavioral profile: how often it changed the response, whether it shifted tone or contributed content, whether the model merely echoed it back without genuine engagement. That profile feeds back into retrieval scoring, creating a loop where facts that demonstrably matter surface more often and facts that don't gradually fade.

### The two measurement channels

Influence measurement operates through two independent channels, each good at what the other cannot do.

**Channel 1: Chaperone (perplexity delta).** A local quantized 4B-parameter open-weight language model, loaded via MLX, runs two forward passes per injected fact: one with the fact in context, one without. The per-token cross-entropy loss is compared across both passes, producing three scalars (variance delta, spectral flatness, max spike). If any of these exceeds its configurable threshold tuned during calibration, the chaperone "fires" — the fact genuinely shifted the model's output distribution. The chaperone's cross-family accuracy ceiling is 71.2% (frontier reasoning model → Qwen 4B, confirmed across 156 annotations). Same-family (Qwen → Qwen) achieves 98% informational and 88.5% relational precision on clean synthetic data. The cross-family ceiling is structural: tonal shifts are family-specific in probability space, so a Qwen chaperone cannot reliably detect relational influence from a different-family generator. This ceiling is what motivated Channel 2.

**Channel 2: Embedding drift (directional projection).** A local quantized embedding model, loaded via MLX, embeds both the production response (with facts injected) and a fact-free baseline response (generated by the chaperone model on the same user query). The drift vector — `embedding(with_facts) - embedding(baseline)` — is projected onto three pre-computed axes loaded from a calibration artifact: relational, informational, and echo. Each projection yields a scalar. The relational axis points toward warmth, empathy, and open-ended tone; the informational axis points toward specific content engagement; the echo axis points toward sycophantic utility-mirror responses where the model merely acknowledges a fact without genuine engagement. The axes are approximately orthogonal, measuring genuinely different things. Cross-family validated: d=1.76 relational and d=0.64 informational on Llama → Qwen data where perplexity scored 0% relational. The embedding model reads the response text, not the generator's probability landscape, so drift classification works regardless of which model generated the response.

The two channels compose in a combiner (see §4 for the full pipeline trace). The chaperone answers "did this fact matter?" The drift channel answers "how did it matter — relationally, informationally, or as an echo?" Each fact receives both signals per evaluation. The combiner classifies the turn as RELATIONAL, INFORMATIONAL, ECHO, or NOISE based on drift projections, then attributes individual facts within that turn based on whether the chaperone fired for each one.

### The Continuum Memory System

Per-fact behavioral fingerprints accumulate in a CMS table. Nine `dc_*` columns track: four class counts (relational, informational, echo, noise), three projection running means (per axis), a denominator (eval count), and a timestamp. Each chaperone evaluation increments the appropriate class count and updates the running projection means via an incremental mean formula. Seed-fact corpus entries are permanently exempt from CMS writes — they accumulate evaluations for research purposes but their CMS counters are never incremented. See §6 for the full invariant.

The eval cap locks ratios at a configured threshold of accumulated evaluations. Once a fact reaches that count, the chaperone worker skips it entirely on subsequent traces — no forward passes, no drift compute, no CMS write. The fact's behavioral profile is considered converged. It still participates in retrieval with its locked ratios.

### How retrieval uses accumulated signal

Composite scoring combines weighted contributions from BM25 relevance, freshness, confidence, and usefulness, with a CMS boost added on top. The boost activates for facts whose eval count crosses a fallback threshold — enough evaluations for the class ratios to be meaningful. A query classifier categorizes the current user query into one of seven types (code, health, creative, personal, decision, factual, general). The CMS boost routes accordingly: personal/creative/health/decision queries boost facts with high relational ratios; code/factual queries boost facts with high informational ratios; general queries use an aggregate influence ratio across both channels. The routing is exhaustive — all seven query categories are handled.

Echo-heavy facts receive an additional negative penalty governed by an echo-suppression coefficient, applied inside the dual-channel boost function proportional to the fact's echo rate. This prevents echo-dominated facts from accumulating composite-score weight even when their eval count has crossed the boost-activation threshold.

## §2. Integration shapes

Helios supports two production-validated integration shapes. Both run from the same codebase. Neither is deprecated.

The distinction matters because each shape places different responsibilities on the consumer. A Shape A consumer manages helios's transaction boundaries, threading model, and GPU lock directly. A Shape B consumer makes HTTP requests and never touches any of that — the server owns it internally. Every integration contract in §5 is scoped to one or both shapes; the scoping determines which bugs a consumer can hit.

### Shape A: In-process Python library

The consumer imports helios modules, instantiates store classes against a shared SQLite connection, and runs the chaperone worker inside the consumer's own process. The harness is the production instance of this shape: the agent runtime constructs the retrieval, CMS, summary, and chaperone-worker components as imported classes, all sharing one SQLite connection per agent.

The consumer's responsibilities under Shape A:

Transaction discipline applies. Every store method that writes is classified as Category (a) — does not commit internally, participates in the caller's transaction boundary — or Category (b) — commits internally because it runs on its own schedule. The practical consequence: any standalone call to a Category-(a) store method must be wrapped in an explicit transaction context or the caller holds the WAL writer lock indefinitely, starving every other writer in the process. Two violations of this rule surfaced during initial integration and produced cascading "database is locked" failures (fixed in subsequent commits). The rule is simple but the failure mode is misleading — the error appears on the *starved* code path, not the lock-holding one.

Background-thread writers must use a dedicated SQLite connection, not the main async loop's connection. Python's sqlite3 module has per-connection transaction state that is not serialized across threads even with `check_same_thread=False`. The harness's classification callback — invoked from the chaperone worker thread — uses a dedicated callback connection, separate from the main connection. Sharing the main connection produced obscure thread-state errors followed by cascade lock failures (fixed in a subsequent commit).

The chaperone worker constructor must receive a per-agent trace-log path. Without it, the worker falls through to a module-level legacy default path — correct for Shape B's standalone default, wrong for any multi-agent Shape A consumer that needs per-agent trace isolation.

GPU lock must be respected. Both MLX models (chaperone + embedding) share unified memory on Apple Silicon. A module-level threading lock serializes all Metal command buffer encoding. The consumer's process inherits the lock via module import; any additional MLX work in the consumer's process must acquire the same lock.

Schema ownership: helios owns its tables, the consumer owns its tables, both live in the same database file. Schema DDL for a table is authored solely by the module that owns it. Helios migrations run first, then the consumer adds its own tables.

Key interfaces for Shape A consumers: hybrid retrieval (BM25 + dense fusion), facts CRUD + embedding storage, the CMS store (ratios for injection-time boost), the summary store, the chaperone worker (background evaluation thread), the combiner (if the consumer runs its own eval loop), and the central config module (all thresholds and constants).

### Shape B: Standalone HTTP server

Helios runs as its own process. A FastAPI surface exposes endpoints for recall, capture, search, stats, lookup, and health. Consumers call via HTTP. The server owns its own connection pool, transactions, chaperone worker lifecycle, and GPU lock — the consumer manages none of these.

The module-level legacy trace-log default path provides a sensible default for this shape. Standalone callers don't pass an explicit path because the default fires. This is not vestige — it is Shape B's affordance.

Shape B's consumer responsibilities are minimal: JSON content type on all requests, agent identification via path parameter or header, and pre-flattening of multimodal content to plain text before calling capture endpoints (the chaperone input contract requires plain strings at both query and response — see §6). The server handles everything else.

The tradeoff between shapes is coupling versus performance. Shape A eliminates the HTTP round-trip and enables atomic multi-store transactions (turn capture can touch helios tables and consumer tables in one explicit transaction). Shape B provides language-agnostic access, process isolation, and a clean versioning boundary — helios and consumer can ship on independent release cycles. Shape A is the correct choice when the consumer is a Python application willing to import helios and respect its invariants. Shape B is the correct choice when the consumer isn't Python, prefers loose coupling, or doesn't want to manage helios's threading model.

A previous TypeScript-based ContextEngine consumer (now retired) was the original Shape B integration. The harness replaced it as a Shape A consumer. The FastAPI server code remains fully functional for any third-party consumer that wants to integrate helios without adopting Python-native wiring.

## §3. The seven components

The harness is structured as seven components. Helios — the memory library — is component #2. The remaining six are harness-owned infrastructure that consumes helios. This decomposition reflects the current codebase, not earlier design documents that used different labels for components 3 and 4.

Understanding which component owns which responsibility matters for two reasons: it determines where a bug lives when something breaks (helios or harness?), and it determines which integration contracts from §5 apply to a given code change.

### 1. Identity layer

The identity layer assembles the always-on context that sits at the top of every system prompt: a relational-character file, a role-and-constraints file, an awareness-of-other-agents file, a knowledge-about-the-user file, a time-awareness template, and any skill files. These are plain Markdown files on disk in the agent's runtime directory, read from the filesystem on every turn — no database involvement, no caching. The assembler reads them; nothing writes to them at runtime.

Thread context: async path (called during context assembly). Reads: agent's identity filesystem. Writes: nothing.

The identity layer is where facts land in the model's context. Helios facts inject at the identity level, alongside the character file — not as user-message content. This placement is load-bearing: facts at the message level destroy relational tone because the model reads them as conversation rather than known truth (see §5, C8).

### 2. Helios memory

The durable long-term memory substrate. Imported as an editable Python install. Owns five SQLite tables in the per-agent database: `facts` (structured fact storage with embeddings), `response_embeddings` (per-turn response vectors for drift computation), `fact_cms_scores` (the nine `dc_*` columns described in §1), `session_summaries` (narrative summaries over turn batches), and `traces` (per-turn evaluation records).

Helios decomposes internally into its own set of subcomponents:

**Hybrid retrieval** performs BM25 + dense + RRF fusion retrieval with CMS boost. Reads from `facts` via FTS5 and embeddings, reads CMS ratios from `fact_cms_scores`. Runs on the async path during assembly.

**Facts store** handles fact CRUD, embedding storage, and composite scoring. Called from the async path for reads and from the chaperone worker thread for writes during fact capture.

**CMS store** manages dual-channel CMS counters. The writer is the chaperone worker thread; the reader is the async retrieval path. Category (a) — does not commit internally.

**Chaperone worker** is the background evaluation thread. Dedicated OS thread, GPU-locked. Processes enqueued facts: per-fact perplexity evaluation, per-turn baseline generation, drift computation, combiner classification, CMS writes, callback invocation, and JSONL trace logging. This is the component that turns raw fact injection events into accumulated behavioral fingerprints. The full pipeline trace is in §4.

**Combiner + drift + axes** are pure functions plus an axis loader. Drift axes load once from a per-agent calibration artifact at worker startup; the combiner classifies turns; drift computes projections. All called from the worker thread.

**Daemon model holder** is technically harness-side but exists to serve helios: it holds the shared chaperone model and embedding model, both loaded at daemon boot, both serialized by the GPU lock. The models are passed into helios components at construction time.

**Capture orchestration** extracts facts from a completed turn, writes a trace row, and enqueues facts for chaperone evaluation. Invoked from the harness's turn-completion hook. This is the entry point from harness into helios's evaluation pipeline.

Thread context: retrieval runs async; evaluation runs on the dedicated worker thread; capture orchestration is called from the async path but enqueues work for the worker thread. Reads: all five owned tables. Writes: all five owned tables.

### 3. Turn queue

A durable per-turn capture queue that decouples turn completion from summarization. When a turn completes, the turn-complete hook inserts a row into a `turn_queue` table with the conversation ID and turn metadata. The queue-driven summarizer (component #4) drains rows asynchronously, outside the turn's critical path. This guarantees that summarization can fall behind without blocking the user's next turn, and that no turn data is lost if the daemon restarts before summarization completes — undrained rows persist in SQLite.

Thread context: the async path enqueues (inside the turn-complete hook); the summarizer service consumes. Reads: undrained rows. Writes: insert on turn completion, update on drain.

Earlier design documents labeled this component "continuous summarizer" and paired it with a "live-context snapshot." The current implementation splits differently: the turn queue is a durable buffer, and the summarizer is the consumer. There is no separate live-context snapshot component in the current codebase.

### 4. Queue-driven summarizer

Generates narrative summaries over batches of raw turns. Drains the turn queue, reads the corresponding messages from the message store, and calls a dedicated summarization engine (a separate inference instance) to produce prose summaries. The summarization engine is a **separate** instance from the primary inference engine — rate-limit isolation so summarizer traffic doesn't compete with user-facing turns for the same provider quota.

Summaries write to `session_summaries`, a helios-owned table. The summarizer runs as an async task, draining outside the protected turn window — it never blocks a user's in-flight turn. State tracking uses a `summarizer_state` (harness-owned table) to record drain progress.

Thread context: async task. Reads: `turn_queue`, `messages`. Writes: `session_summaries` (helios-owned), `summarizer_state` (harness-owned).

### 5. ContextAssembler

The single assembly interface: given a conversation, query, and token budget, returns an assembled context. Combines identity files (component #1), helios facts and summaries (component #2), conversation messages, and any open spans (component #6) into a token-budgeted context window ready for the inference engine.

Assembly invokes the helios layer with a calibrated fact-limit and summary-limit. The fact-limit matches the chaperone's calibration regime — all thresholds were tuned against a fixed number of injected facts per turn. Inflating this number saturates the chaperone (see §5, C6).

The assembler implements a pruning tier ladder for budget management: tier 0 (nothing dropped), tier 1 (summarized traces dropped), tier 2 (raw and pinned traces dropped with a synthetic marker indicating context pressure and a pending summary). Forward-pinning via open spans prevents pruning of in-flight subagent context. Three recovery layers compose when pruning is needed: helios facts for historical context, continuous summaries for recent context, and live conversation messages for the immediate window.

The assembled context's helios-facts field carries the fact dicts the model actually saw. This field is cached per-conversation and later threaded to the capture path as `injected_facts` — the link between "which facts were injected" and "which facts get evaluated by the chaperone." Breaking this link silently disables the entire evaluation pipeline (see §5, C2).

Thread context: async path (called per-turn before engine invocation). Reads: identity filesystem, helios facts + summaries, message store, spans store. Writes: nothing — pure read pipeline.

### 6. Spans

Pin in-flight subagent dispatches so the assembler doesn't prune their context mid-operation. When the harness dispatches a subagent task (via the spawn-agent tool), it opens a span row in the `spans` table with the conversation ID and dispatch metadata. The assembler's pruning logic checks for open spans and preserves their associated context. When the subagent returns its result, the span closes.

Thread context: async path (open on dispatch, close on result); assembler reads during pruning. Reads: open spans by conversation. Writes: span rows (open/close lifecycle).

### 7. Tier 2 hard-truncate

Last-resort pruning when no amount of summarization fits the token budget. Drops raw and pinned conversation traces and inserts a synthetic marker indicating context pressure and a pending summary, so the model knows context was lost and a summary is in flight. Designed for one-turn degradation with recovery on the next turn via the summarizer catching up.

Thread context: async path (inside assembly). Reads: pruning plan state from the assembler. Writes: synthetic marker into the rendered message list.

## §4. The dual-channel pipeline — end-to-end turn trace

This section walks four complete turn traces through the system: a happy path where facts are injected, the chaperone fires, and CMS counters update (Trace A); an error path where the capture gate holds and prevents bogus signal from entering the CMS (Trace B); a subagent dispatch lifecycle showing how spans pin context while a child agent runs (Trace C); and the pruning ladder that decides what to drop when the token budget is exceeded (Trace D). Together, the four traces define the system's runtime behavior — a new consumer wiring helios in can read these traces to understand what happens, in what order, on what thread, and what breaks if a step is skipped.

### Trace A — happy path

Reference trace: a production trace from initial post-fix validation. Mixed chaperone outcomes, turn classified `relational`. (This trace predates the fact-limit correction; current production injects the calibrated number per §5 C6.)

**Entry.** An inbound message arrives via an external messaging channel or CLI and reaches the agent runtime's turn-handler.

**Assembly.** Inside a Phase-1 transaction (the outer transaction boundary), context assembly is invoked. The helios layer loads with the calibrated fact-limit and summary-limit, calling the hybrid retrieval search. The retrieval path queries two stores: `facts` via FTS5 for keyword matching and via the embedding model for dense similarity, fused through reciprocal rank fusion with CMS boost from `fact_cms_scores`. Session summaries are loaded from `session_summaries`. The assembler returns an assembled-context object containing the rendered prompt and the helios-facts list — the facts the model will actually see.

**Semantic retrieval is live.** The main assembler's helios source is constructed with the loaded embedding model — not a placeholder. This distinction matters: if a placeholder is used, dense retrieval falls back silently to FTS5-only with no error (see §5, C1). The same wiring exists in the subagent path.

**Facts cached for capture.** After assembly, the helios-facts list is cached per conversation. This cache is the link between retrieval and evaluation — the capture path later reads from this cache to know which facts were injected. Breaking this link silently disables the entire chaperone pipeline (see §5, C2).

**Engine call.** The assembled context — with helios facts injected at the identity level — is passed to the inference engine. The failover engine wraps each configured engine in a model-bound wrapper, ensuring that on capability-cascade (e.g., a text-only primary rejecting multimodal input), the fallback engine dispatches its own configured model, not the primary's. The engine streams the response.

**Turn close.** Phase-2 transaction: the message store persists the final message state. The capture function is called with the cached injected-facts list. This is the three-hop threading that was broken during initial integration (fixed in a subsequent commit): assembled-context → cache → injected-facts → capture.

**Helios capture.** The capture path enters helios's orchestration. Two things happen: declarative fact extraction runs (extracting new facts from the assistant's response, independent of chaperone evaluation), and a trace row is written to the `traces` table.

**Chaperone enqueue.** The enqueue predicate evaluates: chaperone worker present, injected list non-empty, response and query non-empty. For Trace A this is true — facts are injected, response and query are populated. Each injected fact is enqueued into the chaperone worker.

**Worker thread processes.** The worker batches facts by trace. The worker runs on its dedicated OS thread, separate from the async main loop.

**Baseline generation.** For each unique user query in the batch, the worker generates a fact-free baseline response. The worker acquires the GPU lock, runs the chaperone model with the user query but no facts in context, and embeds the result. The baseline is cached per-query with an LRU cap. If the same user query was already baselined in this batch, the cached version is reused. Baseline generation takes roughly five seconds — the dominant cost in the pipeline.

**Per-fact chaperone evaluation.** For each enqueued fact, the worker runs two forward passes through the chaperone model: one with the fact in context, one without. This produces per-token cross-entropy comparisons yielding three scalars. If any threshold is exceeded (configurable thresholds tuned during calibration), the fact "fires" with a confidence score; otherwise it's silent with a confidence score. Each eval takes roughly 1.5 seconds (two 4B-model forward passes, GPU-locked).

**Response embedding.** The production response (the model's actual output with facts present) is embedded once per turn — not per fact. This is invariant 6.4 from §6: per-turn embedding prevents the identical-per-fact-projections failure discovered during calibration experimentation, which led to the redesign separating turn-level drift from per-fact attribution.

**Drift computation.** The drift function computes the drift vector: response embedding minus baseline embedding. The projection function projects the drift vector onto the three pre-computed axes loaded from the per-agent calibration artifact. Output: three scalar projections (relational, informational, echo). Sub-millisecond — cosine projections against stored axis vectors.

**Combiner classification.** The combiner performs two-level classification.

Turn-level: first-match cascade on drift projections only:

```
if echo above threshold and rel below relational guard:  ECHO
elif rel above threshold and rel > info:                 RELATIONAL
elif info above threshold and info > rel:                INFORMATIONAL
else:                                                     NOISE
```

Thresholds live on a configurable threshold dataclass with defaults tuned during calibration, overridable through the central config module. The ECHO cascade's relational-guard reuses the relational threshold — there is no separate guard constant. The guard prevents misclassifying a genuinely relational turn with incidental echo signal: ECHO fires only when the response is echo-dominant *and not also relational*.

Per-fact attribution: each fact inherits the turn label if and only if the chaperone fired for that specific fact (above a calibrated chaperone-confidence threshold). Facts where the chaperone was silent are labeled `noise` regardless of the turn label. For the reference trace: turn label `relational`; nine of nineteen facts fired (confidence range 0.26–1.00, inheriting `relational`); ten silent (labeled `noise`).

The chaperone is excluded from turn-level classification — it contributes only at per-fact attribution. Turn labels are derived from drift projections alone. This separation keeps the turn-level signal geometrically clean (one signal source, one coordinate system) while still using the chaperone's per-fact discrimination to prevent relational-credit inflation for facts that happened to be present during a relational turn but didn't cause it.

**CMS write.** The CMS write is wrapped in an explicit transaction context — the CMS store is Category (a), does not commit internally. Without the wrapper, the WAL writer lock is held indefinitely and every other writer starves (fixed in a subsequent commit).

Per-fact updates: class counts incremented based on the fact's attributed class; projection running means updated via the incremental mean formula; eval count incremented. Seed-fact corpus entries are filtered out before any write — their counters are never incremented, preserving their exemption (see §6, invariant 6.1).

**Callback.** A classification callback is invoked from the worker thread. The classification adapter wraps the upsert in an explicit transaction context, where the connection is the **dedicated** callback connection — never the main-loop connection. Using the main connection from the worker thread corrupts per-connection transaction state and produces obscure thread-state errors (fixed in a subsequent commit). The callback is a Shape A integration point: the consumer registers it at chaperone-worker construction, helios calls it from the worker thread, and the consumer must handle threading correctly.

**JSONL trace log.** One JSON line is appended to the per-agent trace-log path passed at worker construction. The JSONL line contains the trace ID, turn label, per-fact decisions, drift projections, chaperone signals, and timing data. This is the primary diagnostic artifact for verifying pipeline health.

### Trace B — error path, capture gate holds

Reference incident: a pre-gate image-turn failure producing garbage drift signals on empty-response turns.

**User sends a multimodal input over an external channel.** The path flows through the channel layer to attachment download to the turn handler with multimodal content. Assembly proceeds normally — assembly returns an assembled context with helios facts populated. The cache is populated.

**Engine cascade fails.** The primary engine (text-only) rejects the multimodal input via a capability-cascade error. A capability error is raised. The failover engine cascades to the multimodal-capable fallback. If the fallback also fails (returns empty content or errors out), the final state is assigned as `errored`, `cancelled`, or `partial`.

**Capture gate fires.** The gate reassigns the injected-facts list to empty before passing it to the capture function:

```python
last_ctx = self._last_assembled.get(conversation_id)
injected_facts = list(last_ctx.helios_facts) if last_ctx is not None else []
# ...
if final_state != "completed":
    injected_facts = []
```

This is a pre-dispatch reassignment, not a conditional skip — the capture function still runs.

**Chaperone skipped.** Capture-from-trace executes — the trace row is still written to `traces` with zero facts injected, and declarative fact extraction still runs (new facts can still be captured from partial responses). But the chaperone enqueue predicate evaluates false: the injected list is now empty, so the predicate short-circuits. No facts are enqueued. No chaperone evaluation runs. No drift is computed. No CMS counters are incremented. No JSONL line is appended. No classification row is written.

This is the correct behavior. Without the gate, the chaperone would evaluate facts against an empty or degenerate response, producing meaningless drift signals. The pre-gate incident produced projection values far outside the normal range (typical relational projections are 0.0–0.4) — because the drift vector from an empty response to a baseline is geometric noise, not a meaningful signal. Those bogus projections would have incremented CMS counters, permanently corrupting the affected facts' behavioral profiles.

The gate was added in a subsequent commit. Its placement — between the cache read and the capture call — means it intercepts the injected-facts list at the narrowest point: after assembly has determined which facts *would* have been evaluated, but before any evaluation machinery is invoked.

### Trace C — subagent dispatch span lifecycle

Reference: a keystone smoke test — a fixture agent dispatched a research subagent via a hosted small-model provider, end-to-end span open → child run → span close.

**Parent invokes spawn-agent.** The spawn-agent tool validates required parameters (task, role, model, output target). A parent trace must already be open — the tool fails loud if not.

**Span opens.** The spans store inserts a row into the `spans` table with the parent conversation ID, parent trace ID, and dispatch metadata. Failure at this step returns an error with no lingering state.

**Assembler pins the span.** On every subsequent assembly call for the parent conversation, a forward-pin computation checks for open spans and preserves the dispatch-to-result context window. The pruning logic (§3 #7) cannot drop traces covered by an open span — so if context pressure arrives mid-dispatch, the parent's in-flight context is pinned regardless of how much other context gets summarized or truncated.

**Child runs with filtered tool registry.** The child subagent is spawned with a role-filtered tool registry constructed via a factory that deliberately excludes spawn-agent itself — a recursion guard. This prevents subagents from spawning their own subagents, a locked invariant.

**Child returns, span closes.** On successful completion, the spans store closes the span row with the child's result. The result returns to the parent as a tool response; the parent's next assembly is free to prune the dispatch context if budget requires it.

**Failure paths.** If the child crashes, the spawn handler catches the exception, logs it, and still closes the span (the "follow-up span close" and "paranoid span close" blocks). A crashed dispatch never leaves an open span — the forward-pin invariant is preserved even under failure.

### Trace D — context pressure and the pruning ladder

Reference: no production trace yet. Unlike Traces A–C, this mechanism has not fired in live traffic — the production agent's conversations have not been long enough to exceed the token budget. The ladder is wired, the classifications populate on every completed turn, and unit tests exercise tier 1 and tier 2 behavior, but the machine has never had to choose what to drop under real pressure. This trace describes the mechanism as implemented and unit-tested; production validation is pending.

**Where context pressure surfaces.** Context assembly builds the rendered context for the next turn by combining identity, helios facts, summaries, and conversation messages within a token budget. When the token cost of the candidate context exceeds the budget, the assembler invokes the pruning planner to decide what to drop. The planner is a pure function — given current state (traces, classifications, pinned spans, summarization progress), it returns a pruning plan describing which traces fall out of the rendered view. The underlying messages stay permanently in the message store; pruning operates on the rendered context only (a consequence of the single-DB, living-document model).

**Mode branch.** Two planner functions exist, one per agent mode. The companion mode (the production agent) uses the classification-aware three-tier cascade. The agentic mode uses oldest-first, ignores classifications, has no tier distinction, no synthetic marker. The mode difference is not a flag on a single function; it is two separate implementations, selected by agent configuration. This is a design commitment from the original component sketch ("companion mode uses the full turn-classification pruning ladder, agentic mode uses simple recency") realized as two code paths.

**The six-band priority ladder:**

```python
PriorityClass = Literal[
    "incomplete",  # partial / cancelled / errored; below noise
    "noise",
    "echo",
    "info_low",
    "info_high",
    "relational",
]

_PRIORITY_RANK: dict[str, int] = {
    "incomplete": -1,
    "noise": 0,
    "echo": 1,
    "info_low": 2,
    "info_high": 3,
    "relational": 4,
}
```

Lower rank drops first. `incomplete` at -1 is below `noise` because errored or cancelled turns carry no signal worth preserving — they failed to produce a completed response, so there is nothing to retain. This is the same gate as §5 C7's capture-gate, applied now at context-time rather than at capture-time: the system refuses to treat failed turns as carrying meaning.

**Priority derivation.** The priority-class function reads two inputs: the classifications table (written by the combiner callback, see below) and the set of incomplete trace IDs (computed from message state). Logic:

- If the trace ID is in the incomplete set → `"incomplete"`.
- Else if no classification exists for the trace → `"noise"` (safest assumption; an unclassified trace is treated as carrying minimum signal).
- Else if turn label is `"noise"`, `"echo"`, or `"relational"` → that label directly.
- Else (informational) — split on attribution ratio: less than half of injected facts attributed → `"info_low"`; else `"info_high"`.

The informational split is the only derived class. Relational and echo and noise come directly from the combiner's verdict; informational splits at retrieval time based on how many of the injected facts actually mattered.

**Sort key:** `(priority_rank ASC, position_in_traces_list ASC)`. Lower priority drops first; within a priority band, older traces drop before newer ones. Position is computed once and reused across sort calls — the input traces list is already in chronological order, so index equals age rank.

**The three-tier cascade.** Traces are partitioned into three buckets by whether they are pinned (open span forward-pin) and whether their conversation-queue row has been drained by the summarizer (a marker advanced past it). Traces fall into exactly one bucket: pinned, summarized + unpinned, or raw + unpinned.

**Tier 1 — drop summarized + unpinned** (lossless). Iterates the summarized-unpinned bucket in sort order (lowest priority, oldest first) and drops traces until total cost fits the budget. The session summary in `session_summaries` carries the substance forward, so pruning the raw trace loses no information the model can detect. If tier 1 alone fits the budget, pruning stops here.

The "summarized" predicate compares each trace's queue-row ID against a summarized-through marker, advanced by the summarizer at batch-drain time. Single writer: a state-store advance method, called from inside a transaction that wraps the summary write, the queue mark-processed call, and the marker advance atomically — either all three land or none do. A summarizer crash mid-batch leaves the queue rows unprocessed and the marker un-advanced; the next service pass re-drains them. Reader: context assembly reads the current value and threads it into the pruning input, where the summarized-predicate function partitions traces into summarized-unpinned (tier 1 target) vs raw-unpinned (tier 2 target). This is the second "one writer, two readers" chain in the pipeline — the first being the classifications table described at the end of this trace.

**Tier 2 — drop raw + unpinned** (lossy). If tier 1 exhausts the summarized-unpinned bucket without fitting budget, the planner falls through to raw unpinned traces. These have no summary coverage yet (the summarizer hasn't caught up), so dropping them loses substance. When this tier fires, the synthetic marker is inserted (see below).

**Tier 3 — drop pinned** (last resort). If both tier 1 and tier 2 are exhausted and budget is still exceeded, the planner drops pinned traces. This should essentially never fire — it means open spans are holding more context than the budget can accommodate. When it fires, something is wrong with span management or the budget is too tight for the workload.

**Synthetic marker.** A synthetic context-pressure marker is constructed indicating the number of turns condensed, the wall-clock time, and that a summary is pending. Guarded so it fires only when tier 2 or tier 3 dropped something — never when only tier 1 (lossless) pruned. This gives the model a coherent story for the context gap: substance was dropped, a summary is on the way, proceed with what remains.

**Agentic mode diverges.** No classification read, no priority ranking, no tier partition. Single loop: iterate traces in chronological order, drop oldest unpinned traces until budget fits. No synthetic marker — agentic sessions are task-focused, and a marker acknowledging lost relational substance would be out of register. The docstring states the invariant explicitly: *"Agentic mode: oldest-first, ignore classifications, respect pins."*

**The shared substrate.** The classifications table is the mechanical realization of §0's living-document thesis. One writer, two readers:

- **Writer:** the combiner callback constructed by an adapter factory invokes the classifications-store upsert inside an explicit transaction context (§5 C5). The store's upsert issues a single INSERT-OR-REPLACE — no other code path writes to the table.
- **Reader 1: CMS boost at retrieval.** The dual-channel CMS-boost function reads `turn_label` indirectly via the per-fact CMS class-count columns, which the combiner increments based on the same classification decision that populates the classifications table. The effect is that fact-level retrieval weighting inherits the turn-level verdict.
- **Reader 2: pruning priority.** The priority-class function reads `turn_label` (and attribution ratio) directly to compute pruning priority.

The combiner writes once per completed turn. Two systems read that write for different purposes: retrieval decides which facts to inject next time; pruning decides which turns to retain under pressure. This is what §0 means by "the context window is a living document" — not that something is rewritten, but that classification signal flows from the evaluation pipeline into both the retrieval layer and the retention layer through a shared substrate.

**Why this has not yet fired.** The production agent's conversations produce roughly 500–2000 tokens per turn; the configured budget is well above the turn-count × per-turn-cost threshold needed to trigger tier 1. Real context pressure requires either a much longer continuous conversation, a smaller budget, or a burst of long turns. Until one of those arrives, the ladder's production behavior is inferred from unit tests, not observed. Until then, the mechanism is shipped but silent.

### What the four traces demonstrate together

Trace A shows the full pipeline: retrieval → assembly → engine → capture → enqueue → baseline → chaperone → drift → combiner → CMS → callback → JSONL. Every step has a code location. Every integration contract from §5 is exercised. A reader debugging a pipeline failure can walk Trace A step by step, checking each intermediate state.

Trace B shows the protective boundary. The capture gate is a single reassignment, but it silences the entire downstream pipeline by causing the enqueue predicate to short-circuit. The trace row is still written (for auditability), declarative fact extraction still runs (for capture completeness), but no evaluation signal enters the CMS. The system correctly distinguishes between "the model produced a response and we should measure it" and "the model failed and we should not pretend we measured anything."

Trace C shows the companion-with-hands mechanics. The span lifecycle is the mechanical guarantee that the parent conversation continues uninterrupted while specialized work happens in isolation. The forward-pin is what makes this safe under context pressure; the recursion guard is what makes it finite; paranoid span closure on failure paths is what makes it reliable.

Trace D shows the retention half of the living-document thesis. Classification signal written by the combiner once per turn is read by two systems for different purposes: CMS boost decides which facts to inject next time; the pruning ladder decides which turns to keep when budget is tight. The classifications table and the summarized-through marker are the two durable substrates that let context be "rebuilt fresh each turn from durable storage" rather than handed forward as session state. The mechanism is wired and unit-tested; real context-pressure validation awaits a long-enough conversation. When it arrives, the first tier 2 firing will leave a synthetic marker in a conversation log — the visible evidence that pruning crossed from lossless into lossy territory.

## §5. Integration contracts

A contract is a rule the consumer must follow when wiring helios. Violating a contract does not produce an error at the call site — it produces silent degradation, corrupted data, or cascading failures elsewhere in the pipeline. The contracts are numbered C1–C13 for cross-reference; two convention-only contracts follow.

Each contract is scoped to Shape A (in-process library), Shape B (standalone HTTP server), or both. Shape B consumers are shielded from most of these because the server owns its own transactions, threading, and GPU lifecycle. Shape A consumers — including the harness — must respect all thirteen.

### C1 — The helios source must use a loaded embedding model

**Scope:** Shape A. **Rule:** when constructing the helios source for the assembler, pass the already-loaded embedding model, not a placeholder. **Reason:** dense retrieval falls back silently to FTS5-only if the embedding model is unloaded. No error is raised; retrieval simply degrades to keyword matching, losing semantic recall. **Failure signature:** the trace row records zero facts injected even when FTS5 hits exist; silent semantic-recall loss with no log output. **Historical violation:** fixed in a subsequent commit during initial integration.

### C2 — Injected-facts must carry real fact IDs

**Scope:** Shape A. **Rule:** the capture function must receive the live helios-facts list, sourced from the cached assembled-context object. **Reason:** the chaperone enqueue predicate requires a non-empty list. Hardcoded empty silently disables the entire chaperone path — no evaluation, no drift, no CMS updates, no JSONL. **Failure signature:** the JSONL trace log never advances; eval-count never increments; the classifications table is empty for all turns. **Historical violation:** fixed in a subsequent commit during initial integration.

### C3 — JSONL trace path must be per-agent

**Scope:** Shape A. **Rule:** every chaperone-worker construction site passes a per-agent trace-log path. **Reason:** multi-agent deployments would commingle traces. The module-level legacy default is correct for Shape B's standalone default but wrong for Shape A consumers running multiple agents. **Failure signature:** traces land at the legacy default location instead of the per-agent path; per-agent trace analysis produces commingled results. **Historical violation:** fixed in subsequent commits.

### C4 — Background-thread writers must use dedicated connections

**Scope:** Shape A. **Rule:** the classification callback — invoked from the chaperone worker thread — must use a SQLite connection separate from the main async loop's connection. **Reason:** Python sqlite3's per-connection transaction state is not serialized across threads even with `check_same_thread=False`. Sharing the main connection from the worker thread corrupts transaction state. **Failure signature:** obscure thread-state errors on the worker thread, followed by cascade lock errors on the next transaction from the main loop. **Historical violation:** fixed in a subsequent commit.

### C5 — Category-(a) callers must wrap writes in a transaction boundary

**Scope:** Shape A. **Rule:** any standalone caller of a Category-(a) store must ensure a transaction boundary exists above the call site — an explicit context manager, an outer transaction wrapper, or an explicit BEGIN. Python sqlite3 auto-begins on the first write statement; without a matching commit, the connection holds the WAL writer lock indefinitely, and the busy-timeout expires on every subsequent writer. **Reason:** Category-(a) stores do not commit internally — they participate in the caller's transaction boundary. **Failure signature:** "database is locked" errors cascading across writers within the busy-timeout window. The error appears on the *starved* code path, not the lock-holding one, making diagnosis hard. **Historical violation:** two places, fixed identically across both repos in subsequent commits.

### C6 — Fact-limit at all call sites must match the chaperone calibration regime

**Scope:** both. **Rule:** all helios-load call sites use the calibrated fact-limit; the degradation ladder uses the calibrated reduction sequence. **Reason:** every chaperone threshold was calibrated against a fixed injection count. Multiplying that count saturates the chaperone — all facts evaluate at confidence near 1.00 — and blows eval time linearly. **Failure signature:** the JSONL trace log shows identical digests across many facts per trace; per-fact discrimination collapses; eval time per turn balloons. **Historical violation:** fixed in a subsequent commit.

### C7 — Capture gate on `state != "completed"`

**Scope:** Shape A. **Rule:** the injected-facts list is reassigned to empty before the capture call when the final turn state is anything other than `completed`. **Reason:** empty or errored responses produce degenerate drift vectors — observed projection values far outside the normal range during a pre-gate incident. The combiner has no valid classification on a degenerate response, and allowing CMS updates would permanently corrupt the affected facts' behavioral profiles. **Failure signature:** bogus CMS increments on error turns; classification rows for turns that shouldn't be classified; absurd projection values in the JSONL. **Note:** the capture function itself still runs (declarative fact extraction, trace row write); only the chaperone enqueue predicate evaluates false because the list is empty. **Historical violation:** fixed in a subsequent commit.

### C8 — Identity-level injection, not message-level injection

**Scope:** both. **Rule:** facts must inject at the identity layer (top of system prompt, alongside the relational-character file), not as user-message content. **Reason:** identity vs conversation framing. Facts at the message level destroy relational tone — the model reads them as conversation rather than known truth, producing observable tonal collapse. **Failure signature:** observable behavioral change — the agent reads injected memories as if a user said them, not as if it already knows them. Diagnosable only by reading the model's output, not by telemetry. **Historical violation:** pre-harness era; the harness inherits the correct pattern.

### C9 — GPU lock serializes all MLX access

**Scope:** Shape A. **Rule:** every MLX forward pass — embedding, chaperone eval, baseline generation — acquires the central GPU lock first. **Reason:** Apple Silicon Metal cannot handle concurrent command buffer encoding from multiple threads. **Failure signature:** Metal crash or silent corruption. Not directly observable via logs; detectable only as crash or as long-tail evaluation durations during concurrent embedding queries. **Historical violation:** none — architectural invariant from day one.

### C10 — Trace IDs generated by helios, not the consumer

**Scope:** Shape A. **Rule:** the trace store generates UUIDs; the consumer captures and reuses the returned value, never generates its own. **Reason:** helios owns the `traces` table schema; if the consumer generates IDs independently, the foreign-key relationship between traces and downstream tables (CMS writes, JSONL references) breaks. **Historical violation:** resolved during initial wiring (a double-trace fix).

### C11 — Timestamp boundary: helios int-ms, consumer ISO, conversion at one file

**Scope:** Shape A. **Rule:** all conversions between helios's integer-millisecond timestamps and the consumer's ISO-8601 timestamps happen in one adapter file. No sibling file re-implements the conversion. **Reason:** duplicate conversion logic drifts silently. A second conversion with a different epoch or rounding produces timestamps that sort correctly in one store and incorrectly in the other. **Historical violation:** none.

### C12 — Schema ownership: each module owns its own tables

**Scope:** Shape A. **Rule:** helios owns its tables. The consumer owns its own tables. Same DB, same connection, no cross-module schema mutation. Schema DDL for a table is authored solely by the module that owns it. **Reason:** cross-module DDL creates hidden coupling — a consumer migration that alters a helios table breaks helios on the next library update. Helios's trace store was extended in-place to accommodate consumer needs; the consumer did not mutate helios tables.

### C13 — Cross-provider failover requires each engine to know its model

**Scope:** Shape A. **Rule:** every engine constructor return is wrapped in a model-bound wrapper. **Reason:** a bare engine constructor silently discards the model argument. On capability cascade, the fallback engine would dispatch its *default* model (often empty or misconfigured), not the configured one. **Failure signature:** cross-provider cascade fails with provider-side endpoint errors in a retry loop even with working fallbacks configured. **Historical violation:** fixed in a subsequent commit (latent; first surfaced during a multimodal cross-provider cascade).

### Convention-only contracts (unenforced)

Two contracts exist as reviewer discipline rather than code enforcement:

"Default to Category (a) for new stores" — no automated check. When adding a new store class, the reviewer must verify that write methods do not commit internally. Violation produces the same WAL-starvation failure as C5 but is invisible until the new store runs under load alongside other writers.

"When adding a new background-thread writer, audit for transaction wrapping" — no automated check. Any new callback, timer, or worker that writes to a Category-(a) store from a non-main thread must be audited for the C4 + C5 pattern: dedicated connection, explicit transaction boundary. Violation produces the same misleading error cascade (symptom on starved path, cause on lock-holding path) that took hours to diagnose during initial integration.

Both convention-only contracts are candidates for automated enforcement — a pre-commit hook that greps for internal commits in store write methods, or a test that asserts every background writer uses a connection distinct from the main loop's. Neither is implemented.

## §6. Load-bearing invariants

Invariants differ from contracts: contracts (§5) are rules the *consumer* must follow; invariants are properties the *system* upholds internally. A consumer can rely on these properties without enforcing them — but a developer modifying helios internals must preserve them, because violating an invariant corrupts the pipeline silently.

The numbering (6.1–6.11) is stable for cross-reference from other documents.

### Seed-fact corpus permanence

**6.1 — Seed-fact corpus entries are exempt from CMS writes and boost.** The original seed-fact corpus is permanently excluded from CMS counter incrementation at the write path (filter inside the combiner-decisions recorder) and from CMS boost at the retrieval path (exemption in both single-channel and dual-channel boost functions). Seed-fact entries still accumulate chaperone evaluations — they pass through the worker, receive drift projections, and appear in JSONL — but their CMS counters are never incremented. This preserves the seed corpus as a stable retrieval baseline: those facts surface on BM25 + freshness + confidence scoring alone, unaffected by dual-channel signal that was calibrated for auto-captured facts.

### Database and schema

**6.2 — Single DB per agent.** Each agent's entire state — helios tables and consumer tables — lives in one SQLite file. Helios initialization runs first on the shared connection before consumer tables are added. There is no per-module database split. This enables atomic cross-module transactions (turn capture touching both helios traces and consumer messages in one transaction) at the cost of requiring the transaction discipline described in §5 C5.

### Evaluation lifecycle

**6.3 — Eval cap locks ratios at a configured threshold.** Once a fact accumulates the configured number of chaperone evaluations, the worker's scheduling gate skips it entirely — no forward passes, no drift compute, no CMS write. The fact's behavioral profile is considered converged. It still participates in retrieval with its locked ratios. The cap value was tuned during initial production with documented rationale.

**6.4 — Response embedding is captured once per turn, not per fact.** The worker batches facts by trace and embeds the assistant response once. The drift vector is computed per-turn, not per-fact. This is a post-experimentation invariant: the original design embedded per-fact, which produced identical drift projections across all facts in the same trace (since the response embedding was the same for all). The redesign separated turn-level drift from per-fact attribution — turn classification operates on per-turn projections, while per-fact attribution uses the chaperone signal.

**6.5 — Baseline generation is cached per user query.** A baseline cache stores response and embedding pairs keyed by user query, LRU-capped. When multiple facts from the same trace share a user query, the baseline is generated once and reused. This reduces per-turn GPU time from O(N facts) baseline cost to a single baseline plus O(N facts) per-fact eval cost.

### Drift axes

**6.6 — Drift axes are locked at calibration, not recomputed at runtime.** The per-agent calibration artifact is loaded once at worker startup. There is no runtime recomputation path — the axes are treated as fixed coordinate systems for the lifetime of a deployment. Recomputation would invalidate all previously accumulated CMS projections (running means computed against the old axes would become geometrically meaningless against new ones).

**6.7 — Rel/info axes are shared cross-family; echo axis is per-generator pair.** The relational and informational axes transfer across model families — angular separation ~15° cross-family per the calibration experiments. The echo axis does not transfer: it rotates per generator because each model's sycophantic-mirror patterns project differently in embedding space. The production calibration artifact must contain the correct echo axis for the generator it serves; a second agent's artifact was removed when discovered to contain a mismatched echo axis byte-copied from the first agent's pre-fix artifact, and that agent runs on a chaperone-only fallback until a generator-specific echo calibration is performed.

### Threading and GPU

**6.8 — Chaperone runs on a background worker thread, not the async path.** The worker's per-trace processing is invoked from a dedicated OS thread started at worker construction. The async assembly path never awaits chaperone evaluation — the chaperone is fire-and-forget from the async path's perspective (enqueue returns immediately). This keeps chaperone latency (per-fact eval plus baseline generation) off the user-facing response path.

### Chaperone input contract

**6.9 — Chaperone input contract is plain string at both query and response.** The capture path from agent-result content through the persistence phase through capture-from-trace to chaperone-worker enqueue is typed as plain string at every hop. Multimodal content (images, tool-result blocks, structured data) is resolved to text upstream at the agent loop or engine layer before agent-result content is set. No flattening function exists at the chaperone boundary because the upstream contract already guarantees plain strings. This invariant is currently enforced by type annotations only — a future change to agent-result content accepting a union type would silently break chaperone perplexity evaluation, since base64 image data tokenizes without error but produces meaningless perplexity deltas.

### Retrieval calibration

**6.10 — Calibrated fact-limit at all call sites.** All helios-load call sites use the calibrated fact-limit, and the helios-source default matches. The degradation ladder uses the calibrated reduction sequence. This matches the chaperone's calibration regime (all thresholds tuned against the same injection count). See §5 C6 for the contract and failure signature.

### Trace and timestamp authority

**6.11 — Trace ID authority is helios.** The trace store generates trace IDs via UUID. The consumer captures the returned ID and threads it through its own tables (classifications, JSONL references). The consumer never generates its own trace IDs. This maintains foreign-key integrity between helios-owned trace rows and consumer-owned classification rows in the shared database.

## §7. Architectural patterns worth naming

Four patterns recur across helios but are not helios-specific. Each is a general architectural principle with at least one concrete instance in this system. They are documented here because naming a pattern makes it catchable — a developer who recognizes the shape can check for it proactively instead of discovering it through a production incident.

### Calibrated values carry regime-specific assumptions that don't survive copy-paste

A calibrated value — whether it's a scalar threshold, a high-dimensional axis artifact, or a hardcoded constant — encodes assumptions about the context it was calibrated in. When that value travels to a new context via copy-paste, file duplication, or config inheritance, the assumptions don't travel with it. The mismatch is invisible until someone specifically checks, because the system continues running and producing outputs that look plausible.

**Instance 1: echo axis mis-calibration (live 8 days).** A cross-generator baseline experiment bundled an echo axis calibrated for a different model into the production artifact. That file was deployed and byte-copied to a second agent's artifact. Neither deployment verified that the echo axis matched its target generator. The correct echo axis from an earlier experiment sat orphaned in a non-production file, never loaded by any code path. Cosine similarity between the two echo axes: -0.253, nearly opposite directions. The echo detector was measuring drift against the wrong direction for eight days. Masked in production because the echo rate was 0% under the production agent's conversational register — the detector had nothing to fire on, so the wrong axis produced the same result (no detection) as the right one. Discovered during an architecture investigation when artifact provenance was inspected directly.

**Instance 2: fact-limit regression.** Chaperone thresholds were calibrated against a fixed injection count. The harness hardcoded a multiplied count at three call sites. The chaperone saturated — every fact evaluated at near-perfect confidence — because its perplexity discrimination was tuned for the calibrated count. The inflated value was plausible (more facts = more context, a reasonable default), but it carried an implicit assumption that the chaperone's thresholds would still discriminate at higher injection volumes. They didn't.

**The general principle:** calibrated values should travel with their calibration provenance or refuse to load. A concrete mitigation (banked, not yet implemented): production calibration artifacts should carry a metadata field declaring their calibration target, and the loader should verify the declared target matches the agent's configured model at daemon startup. This would have caught the echo bug on day one instead of day eight.

### Smoke-alarm classes

A smoke-alarm class is an architecturally present classification category that is empirically validated but dormant in production because the condition it detects is absent — not because the instrument is broken. The temptation is to tune the detector's threshold downward to "make it fire," or to remove the class entirely as dead code. Both are wrong. The detector is a guardrail against future conditions that haven't occurred yet in the current configuration.

**Instance: ECHO at low rate.** The ECHO class fires when the echo-axis projection exceeds its threshold and the relational projection is below its guard — a turn where the model merely acknowledges injected facts without genuine engagement, and no substantive relational movement is present. The production agent's character + register + register-matched conversational partner produces few utility-mirror responses, so the ECHO rate is very low across hundreds of production traces. The axis is experimentally validated: d=1.30 (Nemotron), d=1.90 (frontier reasoning), d=2.68 (Gemma) across four generator families. The detector correctly separates sycophantic echo from genuine engagement when echoes are present by construction. It has nothing to fire on in current production because the house isn't on fire.

**Revisit conditions** are defined, not open-ended: model swap (different generator may produce more echo-pattern responses), identity prompt change (different character file could shift the agent's relational register), new agent with a different conversational style, or many traces under a new configuration with continued zero echoes firing.

**The general principle:** do not tune a smoke alarm based on the absence of fires. Validate the instrument against synthetic or historical data where the condition is known to be present. If it discriminates correctly there, its silence in production is evidence of system health, not instrument failure. The instrument earns its architectural presence by being ready for conditions that haven't arrived yet.

### Deliberately-dormant features with explicit enablement criteria

A deliberately-dormant feature is code that is built, tested, and deployed but configured to be inactive. The distinction from dead code is that dormant features have an explicit enablement criterion documented alongside the off-switch — a specific condition under which they should be activated, stated before the system is in a position to evaluate it.

**Instance: epsilon-greedy exploration rate.** The exploration rate is set to zero by configuration. It implements an epsilon-greedy exploration hook designed to prevent confirmation-bias feedback loops: without exploration, boosted facts get evaluated disproportionately in favorable contexts (they're injected because they scored high, and they score high because they were injected — the loop reinforces itself). The hook is dormant because activating it before CMS ratios have stabilized would inject noise into cold-start accumulation. The enablement criterion is concrete: a documented threshold of accumulated facts past the boost-activation gate. Below that threshold, activation would introduce noise; above it, the feedback-loop risk becomes real and exploration starts paying for itself. When the criterion is met, the rate bumps to a small positive value and the activation is recorded as a decision record.

**The general principle:** dormant features are not technical debt. They are pre-positioned capabilities with documented activation triggers. The key discipline is stating the enablement criterion at build time — before the system has accumulated enough data to evaluate it — so that activation is a deliberate decision against a pre-committed threshold, not an improvised judgment call under production pressure.

### Implicit-contract invariants deserve explicit documentation

An implicit-contract invariant is a system property that is currently enforced by type annotations, caller discipline, or implementation convention, but where silent violation is possible through routine refactoring. The invariant holds because of how the code *happens* to be structured, not because anything actively prevents it from being violated. When the cost of silent violation is high — corrupted data, weeks-delayed detection, cascading downstream effects — the invariant deserves an explicit decision record that names the property, the enforcement mechanism, and the failure mode.

**Instance: chaperone plain-string input contract.** The capture path from agent-result content through capture-from-trace to chaperone-worker enqueue is typed as plain string at every hop. Multimodal content is resolved upstream. No flattening function exists at the chaperone boundary because the upstream contract guarantees plain strings. The invariant holds because of type annotations — but a strict type checker is the only enforcement, and a future refactor loosening agent-result content to a union type would silently break chaperone perplexity evaluation. Base64 image data tokenizes without error; the chaperone would run, produce outputs, and those outputs would be garbage perplexity deltas that quietly corrupt CMS counters for weeks before anyone noticed.

**The general principle:** this is the meta-pattern that explains why decision records exist around silent-failure modes. The other three patterns in this section are specific shapes (calibrated values, smoke alarms, dormant features). This pattern is about the practice of *finding* such shapes and writing them down before they bite. The architecture investigation that produced this document surfaced the chaperone string contract as a resolution to an open question — the investigation looked for a flattening function, found it didn't exist, and then asked *why* it didn't need to exist. The answer — upstream type discipline — was the invariant. Writing it into a decision record converted an implicit property into an explicit architectural commitment. The cost of writing the record was twenty minutes. The cost of not writing it would have been measured in corrupted CMS data and weeks of forensic debugging.

The four patterns compose. A calibrated value (pattern 1) whose regime assumptions are enforced only by caller discipline (pattern 4) will produce a silent failure when copy-pasted to a new context. A smoke-alarm class (pattern 2) that is also deliberately dormant (pattern 3) could be mistaken for dead code and removed, destroying the guardrail. Recognizing one pattern in isolation is useful; recognizing them in combination is how architectural discipline prevents production incidents.

## §8. What is NOT helios

Helios is the memory library — components described in §3 component #2 and its internal subcomponents. Everything else in the production stack is harness-owned. This section exists to prevent attribution errors: a reader diagnosing a problem needs to know whether the bug lives in the memory library or the harness, and several harness capabilities sit close enough to helios's domain that the boundary is non-obvious.

"Harness-owned" means the code lives in the harness repository, the design decisions are made in the harness's context, and helios has no dependency on the capability — helios can run without it (and does, under Shape B standalone deployment).

**ContextAssembler.** The assembler *consumes* helios (calls hybrid retrieval, reads the summary store) but is not part of helios. Assembly logic — token budgeting, pruning tiers, identity-layer composition, span-aware context pinning — is entirely harness-owned. A Shape B consumer would build its own assembly layer against helios's HTTP endpoints.

**Queue-driven summarizer.** Generates narrative summaries over turn batches and writes to the summaries table (a helios-owned table, but via helios's API). Uses a **separate** summarization-engine instance — rate-limit isolation so summarizer traffic doesn't compete with user-facing turns for the same provider quota. The summarizer's existence is invisible to helios; helios just sees summaries appearing in its table.

**Identity layer.** Character files, role-and-constraints files, skill files. Read from the filesystem per-turn. Helios facts inject *alongside* identity content at the system-prompt level, but the identity layer itself is harness-owned.

**Tool system.** A configurable per-agent tool registry, with sink-policy enforcement via a permission layer. Helios has no awareness of tools — it sees only the text of the model's response after tool calls have resolved.

**Channel transports.** External-channel integrations (chat bot integration, attachments handling, slash commands) and CLI interface. These are input/output surfaces; helios processes the text that arrives through them but has no knowledge of the transport layer.

**Heartbeat.** A per-agent timer that triggers periodic check-in turns. Current local time is auto-prepended to a heartbeat template so the model can self-gate on quiet hours. Helios evaluates heartbeat turns the same as any other turn — it doesn't know the turn was timer-initiated.

**Guardrails and security scanners.** A guardrails wrapper around inference I/O (input/output scanning configurable per-agent). The tool executor scans tool output in redaction mode via sink policies. Pattern-based scanning (injection / secret / PII categories). Helios is upstream of the guardrails — facts are retrieved before scanning, and the guardrails operate on the inference boundary, not the memory boundary.

**Conversation persistence tables.** Conversations, messages, classifications, turn queue, summarizer state, spans, outbound messages — all harness-owned schema, living in the same database file as helios tables. Same DB, same WAL, same connection semantics — but schema ownership is per-module (see §6, invariant 6.2).

**Failover and model-bound engine.** Cross-provider inference cascade with capability detection. The failover logic is entirely harness-owned; helios has no awareness of which model generated the response it evaluates — by design, since the drift channel works cross-family precisely because it reads text, not the generator's internals.

**Attachments pipeline.** Multimodal-input handling — download, staging, TTL-based garbage collection, multimodal content assembly. Helios sees the text that results from attachment processing, never the raw attachments.

The shared boundary is the SQLite database file. Helios and the harness coexist in one database file per agent with interleaved tables, shared WAL, and uniform transaction semantics. Schema ownership (§6, invariant 6.2) and the transaction framework govern the boundary. Everything above that boundary — how turns arrive, how context is assembled, how tools execute, how models are selected — is harness territory.
